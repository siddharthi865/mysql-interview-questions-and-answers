## Question 1. What are the different types of replication in MySQL?

# What are the different types of replication in MySQL?

## Interview Answer

**MySQL Replication** is the process of copying data automatically from one MySQL server (called the **source**) to one or more **replica** servers. It is primarily used for **high availability, scalability, disaster recovery, backups, and read load balancing**.

In modern MySQL (latest versions), replication is based on the **binary log (binlog)** generated by the source server. Replicas read these events and apply them to stay synchronized.

---

# Types of Replication in MySQL

There are several ways to classify MySQL replication:

1. Statement-Based Replication (SBR)
2. Row-Based Replication (RBR)
3. Mixed Replication (MBR)
4. Asynchronous Replication
5. Semi-Synchronous Replication
6. Group Replication
7. Multi-Source Replication

Let's discuss each one.

---

# 1. Statement-Based Replication (SBR)

In **Statement-Based Replication**, the **SQL statement itself** is written into the binary log.

Example:

```sql
UPDATE employees
SET salary = salary * 1.10
WHERE department = 'Engineering';
```

The above SQL statement is sent to every replica.

### How it works

Source:

```sql
UPDATE employees
SET salary = salary * 1.10
WHERE department = 'Engineering';
```

Binary log contains:

```sql
UPDATE employees
SET salary = salary * 1.10
WHERE department = 'Engineering';
```

Replica executes the exact same statement.

---

## Advantages

- Smaller binary logs
- Lower network usage
- Faster replication for simple statements

---

## Disadvantages

Some SQL statements are **non-deterministic**.

Example:

```sql
UPDATE employees
SET hire_date = NOW();
```

or

```sql
INSERT INTO employees(name)
VALUES(UUID());
```

Each replica may generate different values if the statement is not safely replicated.

Therefore SBR is less reliable for such operations.

---

# 2. Row-Based Replication (RBR)

Instead of logging SQL statements, MySQL logs **each row that changed**.

Example

Suppose the table contains

| ID  | Name  | Salary |
| --- | ----- | ------ |
| 1   | Alice | 75000  |

Run:

```sql
UPDATE employees
SET salary = 80000
WHERE id = 1;
```

The binary log stores something similar to:

```
Before:
Salary = 75000

After:
Salary = 80000
```

Replica simply updates that row.

---

## Advantages

- Very accurate
- No problems with NOW(), RAND(), UUID()
- Recommended for most production systems

---

## Disadvantages

- Larger binary logs
- More storage
- Higher network traffic

---

# 3. Mixed Replication (MBR)

Mixed replication combines **Statement-Based** and **Row-Based** replication.

MySQL automatically decides which format is safest.

Example

Simple statement:

```sql
UPDATE employees
SET salary = salary + 1000
WHERE department='HR';
```

Logged as a statement.

But

```sql
UPDATE employees
SET hire_date = NOW();
```

Logged as row events.

---

## Advantages

- Best balance
- Smaller logs when possible
- Uses row logging when necessary

---

## Disadvantages

- Slightly more complex behavior
- Harder to predict the logging format

---

# Setting the Binary Log Format

Check the current format:

```sql
SHOW VARIABLES LIKE 'binlog_format';
```

Possible values:

```text
ROW
STATEMENT
MIXED
```

Set globally:

```sql
SET GLOBAL binlog_format = 'ROW';
```

---

# 4. Asynchronous Replication

This is the **default replication mode** in MySQL.

Flow:

```
Application
     |
     V
 Source
     |
 Write Binary Log
     |
Client gets SUCCESS immediately
     |
Replica copies changes later
```

The source **does not wait** for replicas before acknowledging the transaction.

---

## Advantages

- Very fast
- Low latency
- High throughput

---

## Disadvantages

If the source crashes immediately after committing but before replicas receive the latest binary log events, some committed transactions may be lost on replicas.

---

# 5. Semi-Synchronous Replication

Semi-synchronous replication improves durability.

Flow

```
Application
      |
      V
 Source
      |
Write Binary Log
      |
Wait for one replica ACK
      |
Return SUCCESS
```

The source waits until **at least one replica** acknowledges receipt of the transaction (not necessarily that it has fully applied it) before confirming success to the client.

---

## Advantages

- Less chance of data loss
- Better reliability

---

## Disadvantages

- Slightly slower
- Depends on network latency

---

# 6. Group Replication

Group Replication allows **multiple MySQL servers** to form a fault-tolerant group with built-in consensus and automatic failure detection.

There are two deployment modes:

- **Single-primary mode** (one writable server, others act as replicas)
- **Multi-primary mode** (multiple writable servers)

Example topology:

```
Server A
     |
Server B
     |
Server C
```

All members keep data synchronized using distributed protocols and certify transactions to avoid conflicts.

---

## Advantages

- Automatic failover
- High availability
- Built-in fault tolerance
- Strong consistency mechanisms

---

## Disadvantages

- More complex configuration
- Higher network overhead
- Conflict detection is required in multi-primary mode

---

# 7. Multi-Source Replication

A single replica receives data from **multiple independent source servers**.

Example

```
Source A ----\
              \
               Replica
              /
Source B ----/
```

Useful when consolidating data from multiple systems.

---

## Advantages

- Data consolidation
- Easier reporting
- Centralized analytics

---

## Disadvantages

- Possible conflicts if sources update the same data
- More complex administration

---

# Sample Schema

```sql
CREATE DATABASE interview_db;
USE interview_db;

CREATE TABLE employees (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(100),
    department VARCHAR(50),
    salary DECIMAL(10,2),
    hire_date DATE,
    manager_id INT
);

INSERT INTO employees (name, department, salary, hire_date, manager_id) VALUES
('Alice', 'Engineering', 75000, '2020-01-15', NULL),
('Bob', 'Engineering', 80000, '2019-03-10', 1),
('Charlie', 'HR', 60000, '2021-07-22', NULL);
```

Suppose you execute:

```sql
UPDATE employees
SET salary = salary + 5000
WHERE department = 'Engineering';
```

### Line-by-line explanation

- `UPDATE employees` – Targets the `employees` table.
- `SET salary = salary + 5000` – Increases the salary of matching employees by 5,000.
- `WHERE department = 'Engineering'` – Limits the update to employees in the Engineering department.

**Replication behavior:**

- **Statement-Based:** Logs the SQL statement itself.
- **Row-Based:** Logs the before/after values for each updated row.
- **Mixed:** Chooses the safest format automatically.

---

# Comparison Table

| Replication Type  | What Gets Replicated                                      | Advantages                            | Disadvantages                             | Typical Use                                  |
| ----------------- | --------------------------------------------------------- | ------------------------------------- | ----------------------------------------- | -------------------------------------------- |
| Statement-Based   | SQL statements                                            | Small logs, efficient                 | Issues with non-deterministic statements  | Legacy/simple workloads                      |
| Row-Based         | Changed rows                                              | Accurate and reliable                 | Larger binary logs                        | Most production systems                      |
| Mixed             | Statements or rows                                        | Balanced approach                     | Less predictable logging                  | General-purpose workloads                    |
| Asynchronous      | Transactions without waiting for replicas                 | Fast, low latency                     | Possible replica lag/data loss on failure | Default replication                          |
| Semi-Synchronous  | Waits for at least one replica to acknowledge receipt     | Better durability                     | Higher commit latency                     | Critical systems needing stronger durability |
| Group Replication | Replicated and certified transactions among group members | High availability, automatic failover | More complex setup                        | Mission-critical clusters                    |
| Multi-Source      | Multiple sources to one replica                           | Data consolidation                    | Conflict management                       | Reporting and ETL                            |

---

# Performance Considerations

- **Row-Based Replication** is generally recommended because it safely handles non-deterministic functions and complex statements.
- Use **`EXPLAIN`** to optimize queries before replication, as inefficient queries consume more resources on the source and can increase replication lag.
- Monitor replica lag using `SHOW REPLICA STATUS\G` (look at fields such as `Seconds_Behind_Source`).
- Ensure proper indexing to minimize the execution cost of replicated changes.
- For high availability, consider **Group Replication** instead of manually managing failover.

---

# Common Pitfalls

- Using **Statement-Based Replication** with non-deterministic functions like `NOW()`, `RAND()`, or `UUID()` can lead to inconsistent results.
- Large transactions in **Row-Based Replication** can significantly increase binary log size and network traffic.
- Long-running transactions may delay replicas and increase replication lag.
- Multi-source replication requires careful schema and key design to avoid conflicting updates.
